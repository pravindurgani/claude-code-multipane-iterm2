Foundation

Why This Process Works

Every strength is extracted from real production systems. Every weakness fix is grounded in observed failure patterns across 12 projects.

Pipeline Discipline

3-6 named stages with typed I/O. Predictable, debuggable, independently testable. Appears in 7 of 12 production projects.

Documentation-First

Architecture and intent written before code. CLAUDE.md is a living contract with invariants and hard rules the AI reads at execution time.

Hybrid AI Routing

Right model for right task. Haiku classifies, Sonnet builds, Opus audits. No project relies on a single LLM call succeeding.

Cross-Model Auditing

The model that writes code never reviews its own code. AUDIT pane (Opus, read-only) reviews what IMPL pane (Sonnet) built. Physically separated.

Cost-Conscious Architecture

SHA1-keyed response caching, model routing by complexity, payload caps, Ollama-first for local tasks. 85% API cost reduction achieved.

Security Consciousness

Defence-in-depth: blocklists, AST code scanners, credential stripping, prompt injection mitigation, Docker sandbox, Opus audit gate.

Testing at Scale

2,400+ tests across the portfolio. Smoke tests before implementation, CI before features. Frontend tests alongside backend, not after deployment.

Graceful Degradation

Every feature wrapped in try/except with context logging. If the LLM is down, the pipeline still produces useful output. Nothing crashes delivery.

Upfront Decision

Choose Your Weight Class

Decide in Step 1. Your weight class determines which steps are mandatory, optional, or skippable. This single decision prevents both over-engineering prototypes and under-engineering products.

🔬

Prototype

Will be thrown away or rewritten. Validate the idea fast, don't build infrastructure.

  • ✓ Step 1 — Define problem
  • ~ Step 2 — Arch doc (brief)
  • ✓ Step 3 — Smoke tests only
  • ~ Step 4 — Single LLM provider
  • ~ Step 5 — Quick review only
  • ✗ Step 6 — Skip monitoring
  • ✗ Step 7 — Skip deploy automation
  • ✓ Step 8 — Ship or kill decision
🔧

Tool

Internal, used regularly. Needs to be reliable but audience is known and forgiving.

  • ✓ Step 1 — Define problem
  • ✓ Step 2 — Full ARCHITECTURE.md
  • ✓ Step 3 — CI + full smoke tests
  • ✓ Step 4 — Provider abstraction
  • ✓ Step 5 — Cross-model audit
  • ✓ Step 6 — Heartbeat alerts
  • ✓ Step 7 — Automated deploy
  • ✓ Step 8 — Full triage rules
🚀

Product

External users, runs in production. Zero tolerance for silent failures or data loss.

  • ✓ Step 1 — Define problem + user
  • ✓ Step 2 — Full arch + migrations
  • ✓ Step 3 — CI + frontend + E2E
  • ✓ Step 4 — Multi-provider + caching
  • ✓ Step 5 — Security audit + CLAUDE.md
  • ✓ Step 6 — Sentry + heartbeats
  • ✓ Step 7 — Container + CI/CD
  • ✓ Step 8 — All triage + deadlines
Core Process

Step-by-Step Breakdown

Click any step to expand. Each step builds on the previous — don't proceed until the checkpoint passes.

01
Manual Thinking — No AI
Define the Problem
If you can't explain it without naming a technology, you don't understand it yet.
Manual ›
  • Write a 3-sentence problem statement: what exists now, what's wrong, what success looks like.
  • Identify your user and their decision-making context — not just who they are, but what decision your output informs.
  • Decide the weight class upfront: Prototype, Tool, or Product. This determines which later steps are mandatory.
⚑
GATE: Can you explain the problem without mentioning a technology? If your answer involves "I need Streamlit" or "use FastAPI" — stop. You're describing a solution, not a problem. Think more.
02
AI-Assisted
Write the Architecture Doc
Architecture reviewed before a single line of code. "I'll review it later" means never.
AI-Assist ›
  • Create ARCHITECTURE.md with: system diagram (Mermaid), data flow, tech stack rationale, known constraints.
  • Define pipeline stages. Name them. Fix the count. Best work uses 3–6 stages. More than 6 means you haven't found the right abstractions.
  • Create CLAUDE.md with project invariants, file map, and hard rules. This is what the AI agent reads at execution time.

If the Project Involves a Database

  • Define the schema migration strategy now — Alembic, Prisma, or versioned SQL scripts.
  • Add "Never DROP or ALTER tables directly — use migration scripts" as a CLAUDE.md invariant so the constraint is visible to the AI agent at execution time in Step 4, not just at planning time.

Documentation Freshness Rule

  • Every doc must include a Last verified: date in its header.
  • If a doc hasn't been verified since the last major commit, treat it as potentially stale.
  • Link to code rather than duplicating it. If a doc exceeds 10KB, split it into focused files.
Keeps: Documentation-first Fixes #4: Over-documentation
Note: CLAUDE.md created here and never revisited is as bad as no CLAUDE.md. If more than ~30% needs rewriting at Step 5, that signals the architecture shifted — stop and revisit this step.
⚑
GATE: Has someone (you or AUDIT pane) reviewed the architecture before any code is written? If the answer is "I'll review it later" — review it now.
03
CI + Tests First
Set Up the Safety Net
CI is step 3, not step 30. Frontend tests belong here alongside backend tests.
Automate ›
  • Create the project with: pytest configured, ruff in pre-commit, .env.example (never .env in the repo).
  • Write 3–5 smoke tests for the critical path before implementing it. Test pipeline boundaries (input → output), not just internals.
  • If the project has a UI: add a basic startup test now — verify the app starts and main page renders. Use Playwright for Next.js, screenshot comparison for Streamlit.
  • Set up GitHub Actions: lint + test on push. Template a standard ci.yml and reuse it.
Keeps: Testing at scale Fixes #1: Inconsistent coverage Fixes #3: CI/CD gaps Fixes #6: No frontend tests
⚑
GATE: Does CI run and pass on the empty project (including frontend smoke test)? If no — fix CI first. Write zero features on a broken foundation.
04
AI-Delegated with Constraints
Implement the Pipeline
Build each stage as an independent function with typed I/O. Check scope after every stage.
AI-Delegate ›
  • Use Claude Code IMPL pane (Sonnet) for implementation.
  • Build each pipeline stage as an independent function with typed inputs/outputs.
  • Add caching from day one: SHA1-keyed for LLM responses, disk-persisted with TTL.
  • Apply graceful degradation: every try/except logs with context and continues.
  • Extract all hardcoded values (country lists, event names, thresholds) into YAML config files from the start.

LLM Integration (Conditional)

Prototype Weight Class
  • Start with a single provider (Ollama or Claude). Extract the abstraction later only if the project survives past prototype.
Tool / Product Weight Class
  • Implement the BaseLLMClient provider abstraction early. Support Ollama (free, local) + one cloud provider.

Scope Control

After each pipeline stage: Ask "Am I still solving the problem from Step 1, or am I solving a new problem?" If scope has grown — stop. Update ARCHITECTURE.md or cut the new scope. Planning expands indefinitely if unchecked.
Keeps: Pipeline discipline Keeps: Hybrid AI Keeps: Cost-conscious Fixes #7: Stale reference data
⚑
GATE 4a: Does the pipeline produce correct output for the happy path? Don't move to audit until it does.

GATE 4b: Is scope still matching the problem statement from Step 1? If not — update ARCHITECTURE.md or cut scope back.
05
AI-Audited — Cross-Model
Audit Before You Ship
Switch to AUDIT pane (Opus, read-only). The model that wrote the code never reviews it.
AI-Audit ›
  • Switch to AUDIT pane (Opus, read-only).
  • Run the full test suite including frontend smoke tests from Step 3. Fix all failures before audit.
  • Run a security review: hardcoded secrets, path traversal, prompt injection, unescaped HTML.
  • Create SECURITY_AUDIT.md with severity-ranked findings.
  • Review CLAUDE.md — does it still match the code? Update or delete outdated sections. If more than ~30% needs rewriting, that signals the architecture shifted — revisit Step 2.
Keeps: Cross-model auditing Keeps: Security consciousness Fixes #2: Known bugs unfixed
Anti-Pattern: Don't document bugs you won't fix. Either fix them or accept them with a written rationale. "Will fix later" without a deadline means never. MEDIUM findings get a tracking issue with a deadline.
⚑
GATE: Are all HIGH/CRITICAL findings addressed? Is CLAUDE.md still accurate? Both conditions must be true before shipping.
06
Observability
Add Monitoring and Alerting
Monitor for silence, not just errors. Every unattended pipeline must notify on failure.
Automate ›
  • Add structured logging with severity levels: DEBUG / INFO / WARNING / ERROR.
  • For cron-scheduled pipelines: add a heartbeat check — Telegram alert if the pipeline didn't run or produced zero output.
  • For user-facing apps: add basic error tracking (Sentry free tier, or error webhook to Telegram).
Fixes #5: No centralised monitoring
Key Insight: A pipeline that fails silently is worse than one that doesn't run. Monitor for zero output, not just error codes. Silence is a failure mode.
⚑
GATE: Does the pipeline send an alert when you simulate a failure — kill the process, remove input data, revoke an API key? If no — monitoring isn't live, it's decorative.
07
Deployment as Code
Deploy with Automation
Deployment is code, not a manual process. Redeploy from scratch in under 5 minutes.
Automate ›
  • Containerise the backend with a Dockerfile.
  • Define deployment config: vercel.json, railway.json, or docker-compose.yml.
  • Automate the flow: push to main → CI passes → deploy. No manual steps.
Fixes #3: CI/CD gaps
⚑
GATE: Can you redeploy from scratch in under 5 minutes? If no — automate more. A deployment that requires tribal knowledge will fail when you need it most.
08
Rules-Based Iteration
Iterate — Fix, Ship, or Stop
Prevents both "infinite polish" and "documented but never fixed" failure modes.
Rules ›

Triage Rule

When a bug is found, classify it immediately:

HIGH / CRITICAL

Fix now, before the next feature. No exceptions. This is a blocker.

MEDIUM

Tracking issue with a deadline. If deadline passes unfixed — fix it or downgrade to LOW with written rationale.

LOW

Track it. If still LOW after 3 reviews — close as "accepted limitation."

Iteration Rules

  • Re-prompt rule: When re-prompting after audit failure, include the previous audit feedback in the prompt context to prevent duplicate suggestions.
  • Stop rule: When a retry produces the same feedback twice — stop retrying. Ship with a documented limitation. Endless retries signal a problem needing human thinking, not more AI cycles.
  • "Done enough" rule: A project is shippable when: (1) all HIGH/CRITICAL fixed, (2) happy path works end-to-end, (3) monitoring is live, (4) redeploy from scratch works. Everything else is tracked polish.
Keeps: Duplicate feedback detection

Summary

3 Key Principles

The entire process distilled to three rules you can hold in your head.

The Approach

Build data-intensive systems using a pipeline-first architecture where every project is 3–6 fixed stages with typed boundaries. AI is integrated as a primary intelligence layer but never trusted blindly — every LLM output passes through a verification layer (number checks, schema validation, cross-model audit) before delivery. The stack is consistent: Streamlit for UI, pytest for testing, YAML for config, Pydantic for contracts, and graceful degradation everywhere. Documentation comes before code; auditing comes before shipping.

  • 1. Pipeline Discipline Fix the stages. Name them. Never add stages mid-flight. Predictability beats flexibility. Your best projects have exactly 3–6 stages with typed boundaries between each.
  • 2. Trust but Verify LLMs are primary, never sole. Every AI output has a verification layer — numbers checked by Python, schemas validated by Pydantic, code reviewed by a different model than the one that wrote it.
  • 3. Degrade Gracefully Every feature wrapped in try/except with context logging. If the LLM is down, the pipeline still produces useful output. If the cache is cold, it rebuilds silently. Nothing crashes the delivery.
